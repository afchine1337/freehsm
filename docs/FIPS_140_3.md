# FreeHSM C --- Non-Proprietary Security Policy (FIPS 140-3 candidate)

**Module name:** FreeHSM Cryptographic Module (libfreehsm-fips.so)
**Version:** 1.0.0-FIPS
**Module type:** Software, Level 1 (multi-chip standalone, OS bound)
**Target validation:** FIPS 140-3 (ISO/IEC 19790:2012 + 24759:2017)
**Validation authority:** NIST CMVP (US) / CCCS (Canada)

---

## 1. Module overview

FreeHSM is a software cryptographic module that exposes a PKCS#11 v3.2 interface to applications. It is intended for use as a Soft HSM in production environments where a hardware HSM is unavailable but FIPS 140-3 Level 1 conformance is required (e.g. internal PKIs, code-signing servers, regulated SaaS workloads).

The module's *cryptographic boundary* is the single shared object `libfreehsm-fips.so` plus the OpenSSL FIPS provider (`fips.so`) that it dynamically loads at `C_Initialize`. All other software --- including the calling application, the host OS, and the OpenSSL base/default providers --- is outside the boundary.

## 3. Roles, services, authentication

The module supports two operator roles:

| Role               | Identifier in PKCS#11 | Authentication              |
|--------------------|-----------------------|-----------------------------|
| Crypto Officer (CO)| `CKU_SO`              | PBKDF2-HMAC-SHA-256 200_000 iter |
| User               | `CKU_USER`            | PBKDF2-HMAC-SHA-256 200_000 iter |

Authentication is per slot. PINs must be at least 8 characters. After `FHSM_PIN_MAX_FAILED = 5` consecutive failures, the role is locked (`FHSM_RV_PIN_LOCKED`); only `C_InitPIN` (for USER, called by SO) or `C_InitToken` (for both) clears the lock.

Between failed attempts, the module enforces exponential backoff:
`delay_n = min(500 ms * 2^(n-1), 60 s)`. The throttle is persisted in the token file so a process restart does not bypass it.

## 5. Sensitive Security Parameters (SSP)

All SSPs are stored exclusively in the **OpenSSL secure heap** (`OPENSSL_secure_malloc`), which is a single `mlock`-ed `mmap` arena. The arena is sized at build time (default 256 KiB, configurable via `FHSM_SECURE_HEAP_BYTES`). Allocations are returned to the arena via `OPENSSL_secure_clear_free`, which zeroizes the block before release.

| SSP                       | Algorithm                | Storage                | Zeroization trigger    |
|---------------------------|--------------------------|------------------------|------------------------|
| DEK (Data Encryption Key) | AES-256                  | Secure heap (live)     | `C_Logout`, `C_Finalize`, lock |
| KEK derivés du PIN        | AES-256                  | Secure heap (live)     | After token write     |
| Private keys (RSA, EC, PQ)| RSA-2048 to 4096, EC, PQ | Secure heap (live)     | `C_DestroyObject`, logout |
| Audit MAC key             | HMAC-SHA-256             | Secure heap (live)     | `C_Finalize`           |
| KAT plaintexts            | (see §6)                 | Secure heap (transient)| End of KAT             |
| DRBG state                | CTR_DRBG                 | OpenSSL FIPS provider  | Provider unload        |

## 6. Self-tests

### 6.1 Pre-operational self-tests (FIPS 140-3 §7.10.2)

Executed at `C_Initialize`, before any service becomes available:

1. **Software integrity test** (§7.10.2.a): SHA-256 of the entire `libfreehsm-fips.so` is computed and compared to the value embedded at build time in `fhsm_module_integrity_digest[]`. Mismatch → ERROR state.
2. **Hardened DRBG initialisation**: `fhsm_drbg_init` runs the multi-source seed (`getrandom + RDRAND + /dev/urandom + TSC jitter` → SHA-256 conditioner → `RAND_add`). Source unavailability → ERROR. See [`RNG.md`](RNG.md) for the pipeline.
3. **Cryptographic algorithm self-test (KAT)**: every approved primitive (§4) is exercised against a CAVP-derived test vector. The base set lives in `kat/fhsm_kat_vectors.c`; **v1.1.0-FIPS.1 adds** an extended CAVP set in `kat/cavp_extended.c` (2 AES-GCM-256 from NIST SP 800-38D Annex B + 4 HMAC-SHA-256 from RFC 4231). Any mismatch → ERROR.

### 6.2 Conditional self-tests (§7.10.3)

| Trigger                        | Test                                | Implementation |
|--------------------------------|-------------------------------------|---|
| AEAD decrypt                    | Tag verification (constant-time)    | `fhsm_aes_gcm_decrypt` rejects on tag mismatch |
| Signature verify                | Algorithm-specific verification     | Inline in `C_Verify` |
| **Asymmetric key generation (§7.10.2.b)** | **Pair-wise consistency check** : encrypt-decrypt round-trip (RSA), sign-verify (EC, ML-DSA, SLH-DSA), encap-decap (ML-KEM). Failure latches ERROR and refuses to persist the keypair. | `src/fhsm_pairwise.c`, invoked inline by `C_GenerateKeyPair` |
| **RNG output (§7.10.3.b)** | **Continuous test** : RCT (cutoff 6 successive identical bytes) + APT (W=512, cutoff 51) + CRNGT (16-byte block compared between calls). | `src/fhsm_drbg.c` `health_rct_apt` + `health_crngt` |

### 6.3 KAT report

After init, `fhsm_kat_results()` returns the structured pass/fail/duration vector. The same data is appended to the audit log as event `kat_report`.

## 7. Operational environment

The module is validated for the following operational environment:

- OS: Linux ≥ 5.4 with `mlock` permission and `RLIMIT_MEMLOCK` ≥ 1 MiB
- libc: glibc ≥ 2.31 or musl ≥ 1.2
- OpenSSL: 3.x with the FIPS provider installed and listed in `openssl.cnf`
- Filesystem: ext4 or xfs for the audit log (atomic `append` semantics)

Operators must:
1. Ensure the OpenSSL FIPS provider is loaded and validated (`openssl list -providers` shows `fips` with status `active`).
2. Store the token files (`*.tok`) and audit log files (`*.audit.log`) on a filesystem that is not network-mounted (NFS/CIFS can lose atomicity).
3. Run the module under a dedicated UNIX user with `CAP_IPC_LOCK` to allow secure-heap mlock.

## 8. Mitigations of other attacks

| Attack                  | Mitigation                                       |
|-------------------------|--------------------------------------------------|
| Timing channels on PIN  | Throttle is enforced *before* PBKDF2; constant-time tag/MAC compare |
| Cold-boot DEK exfiltration | Secure heap mlock + zeroize on free          |
| Coredump leak           | `prctl(PR_SET_DUMPABLE, 0)` at C_Initialize     |
| Log injection           | Audit emit refuses any byte outside `[0x20, 0x7e]` and `[",\\]` |
| Tamper of audit         | HMAC-chained log + token-derived MAC key       |
| Brute-force on PIN      | Exponential backoff + max-5 lock                 |
| DRBG seed exhaustion    | SP 800-90A reseed thresholds + reseed entry point |
